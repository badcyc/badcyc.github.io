Android开发中的单元测试

Android 开发中的单元测试（暂不涉及 Instrument Test）

测试

种类

  • 单元测试：单元测试是对软件中的基本组成单位进行的测试，如一个过程、一个函数等等。它是软件动态测试的最基本的部分，也是最重要的部分之一，其目的是检验软件基本组成单位的正确性。一个软件单元的正确性是相对于该单元的规约而言的。
  • 集成测试：集成测试是在软件系统集成过程中所进行的测试，其主要目的是检查软件单位之间的接口是否正确。它根据集成测试计划，一边将模块或其他软件单位组合成越来越大的系统，一边运行该系统，以分析所组成的系统是否正确，各组成部分是否合拍。
  • 系统测试：系统测试是对已经集成好的软件系统进行彻底的测试，以验证软件系统的正确性和性能等满足其规约所指定的要求，检查软件的行为和输出是否正确并非一项简单的任务，它被称为测试的 “ 先知者问题 ” 。因此，系统测试应该按照测试计划进行，其输入、输出和其他动态运行行为应该与软件规约进行对比。
  • 验收测试：验收测试旨在向软件的购买者展示该软件系统满足其用户的需求。
  • 回归测试：回归测试是在软件维护阶段，对软件进行修改之后进行的测试。其目的是检验对软件进行的修改是否正确。

V 模型：

Android 中单元测试又分为两种：

  • Local Unit Test：直接运行在 PC 的 JVM 环境中，在没有 Android Framework的依赖或者模拟 Android Framework 的依赖的情况下可以最大缩短执行时间。这里要注意如果涉及到 Android Framework 的东西，依赖都是直接从 android.jar 中获取的，里面的函数都是没有实现的，需要模拟，而且真实环境中的Android Framework 和它有可能不同。
    • 优点：执行快，完全不需要打包安装到手机
    • 缺点：不够真实
  • Instrumentation Test：需要运行在 Android 环境中，当涉及到 Android Framework 层的东西，并且不能很好处理依赖的情况下使用
    • 优点：很逼真，能够真实测试 Android 环境和 UI
    • 缺点：巨慢，编译打包推送安装跑不掉

比重

单元测试的好处

  • 保证系统中每个最小构成单元的正确性
  • 在修改后能够快速验证修改是否成功
  • 减少 Bug
  • 快速定位 Bug
  • 最后一个听起来比较玄学的提高开发效率

单元测试的窘境

  • 觉得花时间，降低开发效率
  • 自认为不会写出错误的单元
  • 已有项目太复杂，改造成本巨大
  • 觉得集成测试会检测出 Bug

JUnit4 基本功能

JUnit4 通过注解来配置测试，所以需要理解下它重要的 注解 和 断言函数 。网上有些文章说 所有测试方法必须以test开头，否则没有测试选项 ，这说明他用的 Junit3，Junit4不需要这样。

  • @Test：这个我们上面已经使用过，被这个注解标注的方法表示一个测试用例，可以被 JUnit 运行
  • @Before：使用该注解的函数会在每个被 @Test 标注的函数调用前执行
  • @After：和 @Before 正好相反，它会在每个函数调用之后执行
  • @BeforeClass：被这个注解标注的函数会在测试类加载之后执行一遍
  • @AfterClass：使用该注解的方法会在类卸载之前调用一次

Mockito 局限性/PowerMockito

Mockito 模拟对象的原理是基于 CGLIB 的，而CGLIB生成代理对象有其局限性，如final类型、private类型以及静态类型的方法不能mock。

Android Framework

有时候，我们的被测试的类或者其依赖的类需要 Context 这一类对象病调用某个方法，但是我们又不涉及到 UI 也即不需要进行仪器测试，所以进行本地测试是最好的，但是如果直接运行测试代码，100%会遇到运行时异常。那么怎么能够让它跑在我们本地的虚拟机上呢？

一个很关键的点：Android SDK 并没有提供真实完整的 Framework 层 API，只是提供了一些接口调用而已，方法体都是直接抛出一个异常，这个使用 Ctrl+左键 点源码是可以看到的（前提是你没有下载这个 SDK 版本的源码）。

刚刚了解了 Mockito ，那么问题还是很好解决的，模拟对象，然后控制函数的返回值就好了。不过有些涉及到具体硬件设备的 API 还是只能老实的跑仪器测试。

另一个情景：涉及到 Activity 跳转，但是还是不需要界面，这时候就需要使用 Robolectric 测试框架了。Robolectric支持单元测试范围从Activity的跳转、Activity展示View（包括菜单）和Fragment到View的点击触摸以及事件响应，同时Robolectric也能测试Toast和Dialog。更多的自行学习。
